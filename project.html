쿼리 101개가 1개가 되기까지

PPE Facility Management System — 가스 측정기 B2B/B2C 전자상거래 플랫폼

문제

하나의 제품이 여러 화학물질을 측정하고, 사용자가 고른 화학물질을 모두 측정 가능한 제품만 걸러야 하는 다대다 관계 검색이었습니다. 초기 구현에서 제품 100여 개를 조회할 때마다 연관된 화학물질을 매번 추가로 조회하면서, 쿼리 수가 101개까지 치솟는 N+1 문제를 만났습니다.

Query count / before → after
→
101 → 1
JOIN FETCH + GROUP BY / HAVING COUNT

해결

JOIN FETCH로 연관관계를 한 번에 가져오고, GROUP BY + HAVING COUNT로 "선택한 화학물질을 모두 포함하는 제품"만 정확히 걸러내는 JPQL을 직접 설계했습니다. 결과적으로 쿼리 수를 1개로 줄이며 응답 속도를 10배 개선했습니다.

배운 점

AI 코딩 도구가 코드를 빠르게 생성해주더라도, JPA의 연관관계 로딩 전략과 쿼리 실행 계획을 이해하지 못하면 성능 문제를 스스로 진단할 수 없다는 점이었습니다. Cursor AI와 Claude를 개발 워크플로우에 적극 활용했지만, N+1 문제의 원인을 짚어내고 올바른 방향을 제시하는 것은 결국 JPA 내부 동작 원리에 대한 이해였습니다.

다시 한다면
  • QueryDSL 없이 순수 JPQL 문자열로 8차원 필터링을 구현해 컴파일 타임 오류 검증이 불가능했던 부분 — 시간이 더 있었다면 QueryDSL로 타입 안정성을 확보했을 것
  • 자주 조회되는 필터 조합에 Redis 캐싱 레이어를 두어 반복 트래픽에 대한 DB 부하를 추가로 줄였을 것

트레이드오프 1 — 데이터

인천대 기축 BAS 시스템에서 자동으로 데이터를 끌어와 바로 모델링에 들어가고 싶었지만, 확보한 웹 기반 데이터에는 부분적인 결손과 오류가 있었습니다. 그대로 썼다면 예측 모델의 신뢰도 자체가 흔들릴 상황이었습니다.

수작업 관제시스템에서 원본 엑셀 데이터 약 3,100개를 하나하나 다운로드해 9년치 시계열 데이터를 직접 재구성했습니다. 신설 연구소의 첫 국가과제였고, 9개 기관이 이 데이터를 신뢰하고 함께 써야 하는 상황이었기에 "빠르게 보여주는 결과"보다 "다른 기관도 검증할 수 있는 데이터"를 우선했습니다.

트레이드오프 2 — 모델

이미지 데이터에 강한 CNN을 시계열 데이터에 적용해보자는 실험적인 아이디어를 제안했는데, 이는 검증된 Linear Regression보다 리스크가 큰 접근이었습니다. 한정된 연구 일정 안에서 안전하게 검증된 모델만 제출할지, 시간을 더 써서 새로운 방법론을 시도할지를 고민했습니다.

결과와 배운 점

에너지 관측이 정확한 구간에서는 예측 정확도가 높게 나왔고, 반대로 예측이 잘 안 된 구간에서는 오히려 R², RMSE, MAPE 등 성능 지표와 다중공선성(VIF), Feature Scaling의 중요성을 더 깊이 이해하는 계기가 되었습니다.

기술적으로 더 완벽한 방법이 항상 정답은 아니며, 지금 이 프로젝트가 감당할 수 있는 리스크가 어디까지인가를 먼저 판단해야 한다는 것을 배웠습니다.

확실한 방법과 실험적인 방법을 함께 두고, 지금 이 상황이 감당할 수 있는 리스크를 저울질하는 방식으로 일합니다. 완벽한 결과보다, 다음에 무엇을 바꿀지 아는 것이 더 오래 남는다고 생각합니다.
