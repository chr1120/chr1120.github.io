01 — Features

세 개의 역할, 세 개의 화면

같은 데이터를 보더라도 구매자·판매자·관리자가 원하는 것은 다릅니다. 역할별로 필요한 기능만 남기고 나머지는 숨겼습니다.

구매자
  • 화학물질·상태·용도·측정방식 다차원 필터링
  • 제품 상세 — 이미지·가격·포인트·연관 화학물질
  • 장바구니 수량 조정, 선택/전체 관리
  • 현금·포인트 이중 결제 시스템
  • 주문 생성·조회·상세·취소
  • Q&A·리뷰 작성/수정/삭제, 도움됨 카운트
  • 견적 요청(단일/다중), 검교정 요청
Why 가스 측정기는 스펙이 아니라 "어떤 화학물질을 잴 수 있는가"로 구매가 결정됩니다. 검색을 가장 먼저, 가장 정교하게 만들어야 나머지 구매 흐름이 의미가 있었습니다.
판매자
  • 제품 등록 — 카테고리·화학물질·가격 입력
  • 제품 수정 및 판매자별 목록 관리
  • 카테고리·가격대(100만/500만 기준)별 필터링
  • 상품명·모델명·카테고리 검색
  • 주문 접수·확인·배송 처리
  • 판매자 전용 대시보드, 견적 관리
Why 판매자는 등록한 제품이 몇 개인지보다 "어떤 가격대의 무엇이 잘 팔리는지"를 더 궁금해합니다. 그래서 목록 화면 자체에 가격대·카테고리 필터를 기본값으로 넣었습니다.
관리자
  • 통계 대시보드 — 상품·주문·포인트·매출
  • 전체 상품 조회·수정·삭제·등록
  • 전체 주문 현황, 상태 변경, 페이징
  • 세션별 포인트 잔액·사용 내역 관리
  • 주문량 TOP 5, 가격대별 상품 분석
Why 운영자는 개별 주문 하나가 아니라 "지금 시스템 전체가 정상인가"를 먼저 봐야 합니다. 그래서 상세 화면보다 대시보드의 요약 지표를 먼저 설계했습니다.

03 — Technical Highlights

구현 과정에서 판단이 필요했던 지점들

기능 나열보다, 왜 이렇게 설계했는지가 남는 세 가지 지점을 골랐습니다.

화학물질 정밀 매칭 검색

다대다 관계 · JPQL

하나의 제품이 여러 화학물질을 측정하고, 하나의 화학물질이 여러 제품에 쓰입니다. 단순 WHERE IN으로는 화학물질 중 하나라도 걸리면 매칭돼 버려, 선택한 화학물질을 모두 만족하는 제품만 걸러내는 로직이 필요했습니다.

Why 이렇게 설계했는가

애플리케이션 코드에서 각 제품마다 화학물질 목록을 순회하며 필터링할 수도 있었지만, 상품 수가 늘어날수록 N+1과 성능 저하가 뻔히 보였습니다. GROUP BY와 HAVING COUNT를 DB 레벨에서 처리하면 데이터가 몇만 건이 되어도 쿼리 한 번으로 끝난다는 점을 우선했습니다.

product_chemical · GROUP BY / HAVING
SELECT p.* FROM product p
JOIN product_chemical pc ON p.seq = pc.product_seq
JOIN chemical c ON pc.chemical_id = c.id
WHERE c.name IN (:chemicalNames)
  AND (:states IS NULL OR c.state IN (:states))
GROUP BY p.seq
HAVING COUNT(DISTINCT c.name) = :chemicalCount
-- HAVING COUNT = 선택 개수 → "모두 포함"만 정확히 필터링

현금 · 포인트 이중 결제 시스템

PointConverter · 환율 로직

구매 금액에 따라 차등 포인트를 지급하고, 포인트로 결제 시 원화로 환산하는 로직이 필요했습니다. 100포인트 = 100,000원, 20만 원 이상 구매 시 230포인트로 추가 할인율을 적용합니다.

Why 이렇게 설계했는가

포인트 환율 계산을 Order·Cart 서비스 여기저기서 각자 구현하면, 나중에 환율이나 할인 정책이 바뀔 때 수정 지점을 놓치기 쉽습니다. 그래서 계산 로직 자체를 PointConverter 하나로 격리해, 정책이 바뀌어도 한 곳만 고치면 되도록 만들었습니다.

util/PointConverter.java
public class PointConverter {
    private static final double RATIO = 1000.0; // 1포인트 = 1,000원

    public static int calculateDiscountPoints(int total) {
        if (total >= 200000) return 230;
        if (total >= 100000) return 100;
        return 0;
    }
}

트랜잭션 기반 주문 처리

@Transactional · 원자성 보장

주문 생성 → 주문 아이템 생성 → 장바구니 초기화는 하나라도 실패하면 전체가 되돌아가야 합니다.

Why 이렇게 설계했는가

주문 저장까지는 성공했는데 장바구니 초기화만 실패하면, 사용자는 같은 상품을 다시 결제하는 사고로 이어질 수 있습니다. 트랜잭션 경계를 메서드가 아니라 "결제 완료라는 하나의 비즈니스 단위"로 잡아, 어디서 실패하든 처음 상태로 되돌아가게 만들었습니다.

service/OrderService.java
@Service @Transactional
public class OrderService {
    public Order createCashOrder(...) {
        // 1. 장바구니 검증  2. 주문번호 생성
        // 3. 주문 생성  4. 주문 아이템 생성
        // 5. 저장 → 실패 시 전체 롤백
    }
}
04 — State Machine

주문 상태 파이프라인

7단계 상태를 열거형으로 관리하며, 상태 전환 가능 여부를 서버에서 직접 검증합니다.

PENDING→ CONFIRMED→ PROCESSING→ SHIPPED→ DELIVERED
↳ 취소 경로 PENDING / CONFIRMED / PROCESSING → CANCELLED ↳ 환불 경로 DELIVERED → REFUNDED
Why 상태를 열거형으로 관리하는가

주문 상태를 문자열로 자유롭게 바꿀 수 있게 두면, "배송중에서 바로 취소"처럼 있어서는 안 되는 전환도 막을 방법이 없습니다. 상태별로 이동 가능한 다음 상태를 코드로 명시해, 잘못된 전환은 서버가 거절하도록 만들었습니다.
